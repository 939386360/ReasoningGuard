# 数据集格式与加载说明

更新日期：2026-06-21

本文档说明本项目提到的数据集如何构成、代码如何加载、需要下载哪些数据，以及下载后应整理成什么本地格式。结论基于当前仓库代码、`config.yaml`、`README.md`、`docs/references/` 中已整理资料，以及 MCPTox/AgentPI 论文页面。

## 1. 最终评估数据集结论

`docs/references/AAAI2027_Agent.pdf` 的实验设置明确写到，最终评估数据集由三部分构成：

|组成|论文中的用途|当前项目落地结论|
|---|---|---|
|MCPTox|T1 tool poisoning 主评估；论文中使用 200 条 scenarios|应使用官方 MCPTox 数据；当前本地已有 `third/MCPTox-Benchmark-main/` 原始数据，并已可适配为 `data/mcptox/mcptox_official.json`|
|AgentPI context-dependent tasks|T2/context-dependent reasoning attack；论文中使用 150 条 scenarios|原论文引用 `Anonymous 2026b / arXiv:2602.10453`，但 PDF 和 arXiv 页面未给出明确下载链接；当前只能使用 synthetic fallback 跑通流程，不能把它当作 official 复现数据|
|MCPTox+|本文扩展数据；80 条 context-dependent scenarios + 50 条 cross-session T3 scenarios|由本项目脚本生成，当前已有 `data/mcptox_plus/mcptox_plus.json`|

因此，正式评估口径应写成：`MCPTox official/adapted` + `AgentPI official if available, otherwise synthetic replacement clearly labeled` + `MCPTox+ generated by this project`。如果目标是论文级复现，AgentPI official 数据仍是缺口；如果目标是本项目流程验证或 synthetic simulation，当前 `load_agentpi.py` 自动生成的 150 条数据可以使用，但结果必须标注为 synthetic。

## 2. 当前项目数据资产概览

|数据集/数据资产|用途|当前本地状态|主要代码路径|
|---|---|---|---|

## 3. MCPTox

### 3.1 数据集构成

外部论文版 MCPTox 是 MCP tool poisoning benchmark。论文描述其基于真实 MCP server 和 tool 构造，包含 45 个 MCP servers、353 个 tools、1312 个 malicious test cases，并覆盖多类风险和三种攻击范式。

但当前项目代码中的 MCPTox 约定更窄：

- `config.yaml` 里配置 `benchmarks.mcptox.num_scenarios: 200`。
- 当前 loader 的 synthetic fallback 生成 200 条场景。
- 四类攻击及 synthetic 数量分布：
  - `tool_description_poisoning`: 55
  - `parameter_injection`: 50
  - `response_manipulation`: 55
  - `capability_escalation`: 40

正式评估应优先使用官方 MCPTox 数据或本地 raw MCPTox-Benchmark 数据适配后的 `data/mcptox/mcptox_official.json`，不建议用 synthetic fallback 作为正式结果来源。当前已经验证 `load_mcptox(use_official=True)` 会读取 `data/mcptox/mcptox_official.json`，返回 1348 条 `source=MCPTox-Benchmark-main` 的 adapted scenarios。

### 3.2 代码如何加载

入口函数：

```python
from src.benchmarks.load_mcptox import load_mcptox

scenarios = load_mcptox(
    data_dir="data/mcptox",
    use_official=True,
    seed=42,
)
```

加载逻辑：

1. 查找 `data/mcptox/mcptox_official.json`。
2. 如果 `use_official=True` 且该文件存在，则读取 JSON 并返回 `data["scenarios"]`。
3. 否则调用 `_generate_synthetic_mcptox(seed=seed)` 生成 200 条 synthetic scenarios。

`src/evaluation/live_table1.py` 会调用 `load_mcptox()`，再通过 `normalize_scenario()` 归一化字段。归一化时会优先读取顶层字段，也会兼容 synthetic 场景里的 `template` 字段。

注意：`src/evaluation/eval_runner.py` 中的 `run_mcptox_experiment(mock_mode=False)` 当前没有调用 `load_mcptox()`，而是直接用 `AttackGenerator.generate_benchmark()` 生成仿真场景。因此要跑官方 MCPTox 真实文件，应优先使用 `src/evaluation/live_table1.py` 这条链路，或先改造 `eval_runner.py`。

### 3.3 本项目可直接读取的官方文件格式

下载到论文原始数据后，不应假设能被本项目直接读取。当前 loader 期望一个包装后的 JSON 文件：

```json
{
  "scenarios": [
    {
      "scenario_id": "mcptox_0001",
      "category": "tool_description_poisoning",
      "temporality": "T1",
      "target_server": "fs-server",
      "attack_vector": "tool_description_poisoning",
      "poisoned_content": "malicious tool metadata or response content",
      "legitimate_content": "benign tool metadata or task content",
      "target_action": "Read and exfiltrate sensitive file"
    }
  ]
}
```

推荐字段含义：

|字段|是否推荐必填|说明|
|---|---|---|
|`scenario_id`|是|场景唯一 ID；缺失时 `normalize_scenario()` 会退回 `original_id` 或 `unknown`|
|`category`|是|攻击类别；建议使用四类之一|
|`temporality`|是|MCPTox 主实验一般为 `T1`|
|`target_server`|推荐|目标 MCP server；缺失时按 category 推断|
|`method`|推荐|目标 MCP method；缺失时按 server 推断|
|`attack_vector`|可选|攻击向量；`load_mcptox.py` synthetic 会写入|
|`poisoned_content`|是|恶意 tool metadata、恶意响应或注入内容|
|`legitimate_content`|是|正常任务、正常参数或正常工具描述|
|`target_action`|是|攻击希望诱导 agent 执行的有害动作|

`normalize_scenario()` 也兼容以下字段别名：

- 恶意内容：`poisoned_content`、`malicious_response`、`template.malicious_response`、`template.poisoned_desc`、`template.injected_param`、`template.claimed_capability`、`template.poisoned_instruction`
- 正常内容：`legitimate_content`、`template.legitimate_response`、`template.legitimate_desc`、`template.legitimate_param`、`template.actual_capability`、`template.legitimate_task`

### 3.4 应下载什么、放在哪里

优先下载 MCPTox 官方数据。论文页面给出的匿名仓库地址是：

- `https://anonymous.4open.science/r/AAAI26-7C02`

下载后需要整理成：

```text
data/
  mcptox/
    mcptox_official.json
```

如果拿到的是论文原始格式，应编写一次性转换脚本，把原始 server/tool/test-case 信息映射到本项目的 `scenarios` 包装格式。不要直接把未知结构的原始 JSON 改名为 `mcptox_official.json`，否则 loader 只要找不到顶层 `scenarios` 就会报错。

### 3.5 本地 `third/MCPTox-Benchmark-main` 适配

本地目录 `third/MCPTox-Benchmark-main/` 中有两个核心 JSON 文件：

```text
third/MCPTox-Benchmark-main/
  pure_tool.json
  response_all.json
  def_tool/
```

只读检查结果：

- `pure_tool.json` 是 poisoned tool 列表，顶层为 45 个 server 分组，共 485 条 poisoned tool 记录。
- `response_all.json` 是更完整的实验/标注数据，顶层包含 `data_length`、`attack_scopes`、`label_scopes`、`call_behaviors`、`servers`。
- `response_all.json` 中 `data_length=1348`，`servers` 覆盖 45 个 server，展平 `servers.*.malicious_instance[*].datas[*]` 后得到 1348 条数据。
- 这两个文件都没有顶层 `scenarios` 数组，因此当前 `load_mcptox.py` 不能直接读取。

验证过的失败路径：

```bash
python -c "from src.benchmarks.load_mcptox import load_mcptox; s=load_mcptox(data_dir=r'third\\MCPTox-Benchmark-main', use_official=True); print(len(s), s[0].get('source'))"
```

结果会回退到 synthetic fallback，返回 200 条 synthetic scenarios。

已新增适配器：

```bash
python -m src.benchmarks.adapt_mcptox_benchmark \
  --input_dir third/MCPTox-Benchmark-main \
  --output data/mcptox/mcptox_official.json
```

适配逻辑：

- 输入：`response_all.json`，可选读取 `pure_tool.json` 统计 poisoned tool 数量。
- 展平路径：`response_all.servers.*.malicious_instance[*].datas[*]`。
- 输出：`{"scenarios": [...]}` 包装格式，放到当前 loader 默认路径 `data/mcptox/mcptox_official.json`。
- 核心字段映射：
  - `scenario_id`: `mcptox_raw_<原始 id>`
  - `benchmark`: `MCPTox`
  - `source`: `MCPTox-Benchmark-main`
  - `category`: `tool_description_poisoning`
  - `attack_layer`: `L4`
  - `temporality`: `T1`
  - `target_server` / `method`: 映射到本项目已有的模拟 MCP server/method
  - `poisoned_content`: 原始 `poisoned_tool`
  - `legitimate_content`: 原始 `query`
  - `target_action`: 由 `security risk` 和 `security_risk_description` 组合生成
  - `metadata`: 保留原始 server、risk、template、模型标签等信息

默认不把 `response`、`system`、`online_result` 等超长字段写入每条 scenario。若后续需要完整追踪原始模型输出，可用：

```bash
python -m src.benchmarks.adapt_mcptox_benchmark --include_responses
```

## 4. AgentPI

### 4.1 数据集构成

AgentPI 是 context-dependent prompt injection benchmark，关注 agent 在依赖运行时上下文和环境观察时被间接注入影响。当前项目代码把它抽象为 150 条 T2/L2 场景。

`docs/references/AAAI2027_Agent.pdf` 中只说明使用了 `AgentPI context-dependent tasks (Anonymous 2026b), 150 scenarios where attacks operate through the reasoning process rather than direct action manipulation`。参考文献条目为 `The Landscape of Prompt Injection Threats in LLM Agents: From Taxonomy to Analysis, arXiv:2602.10453`。PDF 没有给出 AgentPI 数据下载地址、仓库地址或 artifact 链接。

进一步检查 arXiv 页面后，只能确认该论文提出了 AgentPI benchmark，用于评估 context-dependent interaction settings；页面也没有明确列出可直接下载的数据仓库。因此当前不能把项目自动生成的 AgentPI synthetic 数据称为官方 AgentPI 数据。

Synthetic fallback 中使用五类攻击：

- `indirect_injection`
- `context_hijacking`
- `reasoning_redirection`
- `goal_subversion`
- `instruction_override`

每条 synthetic AgentPI 场景包含：

- `scenario_id`
- `category`
- `temporality: "T2"`
- `benchmark: "AgentPI"`
- `source: "synthetic"`
- `setup_context`
- `injection_content`
- `legitimate_task`
- `target_action`
- `difficulty`
- `attack_layer: "L2"`

### 4.2 代码如何加载

入口函数：

```python
from src.benchmarks.load_agentpi import load_agentpi

scenarios = load_agentpi(
    data_dir="data/agentpi",
    use_official=True,
    num_scenarios=150,
    seed=42,
)
```

加载逻辑：

1. 查找 `data/agentpi/agentpi_official.json`。
2. 如果 `use_official=True` 且该文件存在，则读取 JSON 并返回 `data["scenarios"]`。
3. 否则调用 `_generate_synthetic_agentpi()` 生成 synthetic scenarios。

当前主评测脚本没有把 AgentPI official loader 串进完整实验流水线；它主要被测试和潜在扩展使用。

当前实际验证结果：

```bash
python -c "from src.benchmarks.load_agentpi import load_agentpi; s=load_agentpi(use_official=True); print(len(s), s[0]['source'])"
```

输出为 `150 synthetic`。这说明 `data/agentpi/agentpi_official.json` 不存在时，代码会自动生成 synthetic 数据。

### 4.3 本项目可直接读取的官方文件格式

```json
{
  "scenarios": [
    {
      "scenario_id": "agentpi_0001",
      "category": "indirect_injection",
      "temporality": "T2",
      "setup_context": "The user is doing a context-dependent task.",
      "injection_content": "Untrusted content that tries to redirect the agent.",
      "legitimate_task": "Complete the user's intended task.",
      "target_action": "Leak private data to an external destination"
    }
  ]
}
```

推荐字段含义：

|字段|是否推荐必填|说明|
|---|---|---|
|`scenario_id`|是|场景唯一 ID|
|`category`|是|建议使用 loader 定义的五类之一|
|`temporality`|是|AgentPI 在本项目中应为 `T2`|
|`setup_context`|是|正常任务上下文|
|`injection_content`|是|不可信上下文中的注入内容|
|`legitimate_task`|是|用户真实目标|
|`target_action`|是|攻击希望诱导的有害动作|
|`difficulty`|可选|`easy`、`medium`、`hard`|
|`attack_layer`|可选|建议为 `L2`|

### 4.4 应下载什么、放在哪里

若 AgentPI 作者或论文附带官方数据发布，应下载后整理成：

```text
data/
  agentpi/
    agentpi_official.json
```

当前只确认论文页面介绍了 AgentPI benchmark；未在本项目中发现官方数据文件，也未确认一个可直接下载的稳定仓库。因此落地时需要先确认官方发布位置，再做格式适配。

结论：

- 若目标是论文级复现，仍需要找到或向作者获取 official AgentPI 150 scenarios，然后整理为 `data/agentpi/agentpi_official.json`。
- 若目标是跑通本项目流程、做接口联调或 synthetic simulation，可以使用当前 `load_agentpi.py` 生成的 150 条 synthetic scenarios。
- 使用 synthetic AgentPI 时，实验记录和论文表述必须明确标注 `source=synthetic`，不能写成 official benchmark result。

## 6. RTV judge fine-tune 数据

### 6.1 数据构成

这是项目内部合成的监督微调数据，不是需要下载的外部 benchmark。它用于训练 RTV constrained judge，让模型根据 reasoning trace、intent summary、origin tags 输出三类异常分数：

- `CAI`: Context-Action Inconsistency
- `OAV`: Origin-Awareness Violation
- `IAD`: Intent-Action Divergence

默认生成规模：

- benign 样本：500
- 每个攻击类别样本：100
- 攻击类别：6 类，即 `tool_description_poisoning`、`parameter_injection`、`response_manipulation`、`capability_escalation`、`context_dependent`、`cross_session_t3`
- 总样本：1100
- 默认 train/val 比例：0.9/0.1

### 6.2 生成与加载

生成命令：

```bash
python -m src.finetune.judge_dataset
```

生成后目录：

```text
data/
  judge_finetune/
    train.jsonl
    val.jsonl
    meta.json
```

每行 JSONL 样例：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "You are a security judge for an LLM agent that uses MCP tools..."
    },
    {
      "role": "user",
      "content": "Reasoning Trace:\nStep 1:\n  Observation: ..."
    },
    {
      "role": "assistant",
      "content": "{\"CAI\": 0.7, \"OAV\": 0.3, \"IAD\": 0.6}"
    }
  ]
}
```

微调脚本 `src/finetune/finetune_judge.py` 会读取：

- `data/judge_finetune/train.jsonl`
- `data/judge_finetune/val.jsonl`

然后把 `messages` 通过 tokenizer 的 chat template 转成训练文本。

## 7. 推荐下载与整理清单

优先级从高到低：

1. **MCPTox 官方数据**：正式评估应使用官方数据。本地已有 `third/MCPTox-Benchmark-main/` 原始数据，使用 `python -m src.benchmarks.adapt_mcptox_benchmark` 转换成 `data/mcptox/mcptox_official.json`。
2. **AgentPI 官方数据**：当前缺失明确下载地址；若后续取得作者发布的数据，应转换成 `data/agentpi/agentpi_official.json`。在此之前只能使用 synthetic fallback，并明确标注。
3. **MCPTox+**：不下载，使用 `python -m src.benchmarks.build_mcptox_plus` 生成。
4. **RTV judge fine-tune data**：不下载，使用 `python -m src.finetune.judge_dataset` 生成。

最终建议目录形态：

```text
data/
  mcptox/
    mcptox_official.json
  agentpi/
    agentpi_official.json  # 仅在取得官方 AgentPI 数据后存在
  mcptox_plus/
    mcptox_plus.json
    mcptox_plus_stats.json
  judge_finetune/
    train.jsonl
    val.jsonl
    meta.json
```

## 8. 使用注意事项

- `mcptox_official.json` 和 `agentpi_official.json` 必须有顶层 `scenarios` 数组。
- 当前 `load_mcptox.py` 和 `load_agentpi.py` 不校验 schema；字段缺失会在后续评测归一化或 prompt 构造阶段暴露。
- 论文原始数据规模和当前项目 synthetic 配置不完全一致。尤其 MCPTox 论文版是 1312 malicious test cases，而本项目 synthetic fallback 是 200 条场景。
- AgentPI 当前没有接入 official 数据；自动生成的 150 条数据只能作为 synthetic replacement。
- `experiments/run_all.py` 和表格/图生成脚本默认使用 mock 结果，不代表已经读取官方数据重新复现实验。
- 如果目标是官方 benchmark 复现，需要先接入官方文件，再确认评测入口实际调用 loader，而不是调用 `AttackGenerator` 或 mock result。
